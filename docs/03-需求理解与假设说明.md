## 一、分级审批金额阈值

### 假设设定

| 审批层级 | 金额范围 | 说明 |
|----------|---------|------|
| 一级审批 | 金额 ≤ 5000 元 | 常规异常，单级审批即可 |
| 二级审批 | 金额 > 5000 元 | 高金额异常，需双重审批 |

### 设定依据

1. **行业参考**：中小型物流/电商企业中，5000 元通常是一个常见的审批梯度分界线。低于此金额的货损赔付成本可控，单一审批人即可决策。
2. **风险平衡**：假设平均运单金额在 200 元左右，5000 元约等于 25 单货值。超过此量级需要更高级别管控是合理的。
3. **可配置性**：以上阈值为默认值，实际值存储在数据库配置表中，后台可随时调整，不硬编码。

### 配置表设计

```sql
CREATE TABLE approval_config (
  id SERIAL PRIMARY KEY,
  config_key   VARCHAR(50)   NOT NULL UNIQUE,
  config_value VARCHAR(200)  NOT NULL,
  description  TEXT
);
-- 默认数据
-- INSERT INTO approval_config VALUES (1, 'level2_threshold', '5000', '二级审批金额阈值(元)');
```

## 二、审批超时时长

### 假设设定

| 节点 | 超时时长 | 超时后动作 | 计时起点 |
|------|---------|-----------|---------|
| 待审批 (pending_approval) | 48 小时 | 自动升级到二级审批 | 工单创建时间 |
| 一级审批中 (level1_approving) | 48 小时 | 自动升级到二级审批 | 状态变更为一级审批中的时间 |
| 二级审批中 (level2_approving) | 72 小时 | 自动驳回，工单关闭 | 状态变更为二级审批中的时间 |

### 设定依据

1. **响应时效**：以工作日计算，48 小时覆盖 2 个完整工作日，确保审批人有合理的处理窗口。
2. **梯度设计**：
   - 待审批 48h：上报后需要尽快响应，超时意味着没人处理 → 升级。
   - 一级审批 48h：同理，一级审批人不能拖延。
   - 二级审批 72h：到二级审批时已是一级过滤过的，性质更严重，给略长时限但因为更高层级审批人更少、不一定能立即处理，给 3 个工作日。
3. **成本考量**：货压仓/未处理异常每天产生运营成本，48h 是一个合理的平衡点（不太紧不致误操作，不太松不致成本失控）。
4. **注意**：此处是审批超时时长，与品控暂扣超时时长是两套独立配置，后者应远短于前者（见第七节）。

## 三、重新提交次数上限

### 假设设定

| 参数 | 值 | 说明 |
|------|-----|------|
| 最大拒绝次数 | 3 次 | 拒绝 3 次后自动关闭工单 |
| 超过上限动作 | 自动关闭（closed） | 不再接受重新提交 |

### 设定依据

1. **流程闭环**：无限重试会导致工单永远挂在系统中无法关闭。"3 次"是业界常见的审批重试上限（类似密码重试逻辑），既能给上报人足够的机会完善信息，又能防止无限循环。
2. **强制升级**：如果业务需要升级而非关闭，可在配置中切换为"超过上限后自动升级到二级审批"。系统设计上支持两种策略切换，默认采用"关闭"策略（更符合"拒绝意味着不通过"的语义）。
3. **体验考量**：每次拒绝需附带审批意见，让上报人知道如何改进，避免"无意义重试"。

## 四、物流异常类型映射

### 假设设定

| 异常子类型 | 涉及赔付 | 赔付方向 | 库存联动 | 重新发货 | 执行动作说明 |
|-----------|---------|---------|---------|---------|-------------|
| 丢件 | 是 | 赔付客户 | 回滚原运单库存 | 是 | 1. 生成赔付记录 2. 库存回滚（如已出库则标记损耗） 3. 可选重新发货 |
| 破损 | 是 | 赔付客户 | 退货入库（破损仓） | 是 | 1. 生成赔付记录 2. 货物退货入库到"破损区" 3. 重新发货 |
| 客户拒收 | 是 | 赔付客户 | 退货入库 | 否 | 1. 生成赔付记录（视情况减免） 2. 货物退货入库 3. 运单关闭 |
| 超时未签收 | 部分 | 赔付客户 | 退运处理 | 否 | 1. 视服务协议决定是否赔付 2. 货物退运处理 3. 运单标记异常 |
| 收货地址错误 | 否 | 无 | 不需联动 | 是 | 1. 无需赔付（非货物损失） 2. 联系客户更正地址 3. 重新发货 |

### 设定依据

1. **赔付判断**：只有货物本身损失（丢件、破损）或客户已拒收才触发赔付。地址错误属于信息差错，没有货损，不产生赔付。
2. **库存联动**：
   - 丢件：货物"消失"了，需要从库存中扣除。
   - 破损/拒收：货物退回仓库，库存需要"加回"（进入残次品区）。
   - 地址错误：货物仍在，调整收货地址后重新发出即可。
3. **重新发货**：丢件和破损影响客户收货，需重新发货；拒收是客户主动行为，不需要重新发。

## 五、角色权限划分

### 假设设定

| 操作 | 仓库操作员 | 上报人 | 品控主管 | 一级审批人 | 二级审批人 | 管理员 |
|------|-----------|--------|---------|-----------|-----------|--------|
| 扫描录入 | ✓ | ✓ | ✓ | ✗ | ✗ | ✓ |
| 异常上报 | ✓ | ✓ | ✓ | ✗ | ✗ | ✓ |
| 一级审批 | ✗ | ✗ | ✗ | ✓ | ✗ | ✓ |
| 二级审批 | ✗ | ✗ | ✗ | ✗ | ✓ | ✓ |
| 快速放行（品控） | ✗ | ✗ | ✓ | ✗ | ✗ | ✓ |
| 查看自己相关工单 | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ |
| 查看所有工单 | ✗ | ✗ | ✓ | ✗ | ✗ | ✓ |
| 品控规则配置 | ✗ | ✗ | ✗ | ✗ | ✗ | ✓ |
| 审批分级配置 | ✗ | ✗ | ✗ | ✗ | ✗ | ✓ |
| 用户角色管理 | ✗ | ✗ | ✗ | ✗ | ✗ | ✓ |

### 设计原则

1. **一人可多角色**：在数据库层面 `user_roles` 表支持一个用户关联多条角色记录。但业务规则上需有约束：
   - **同一运单的审批链上，一个人不能既是上报人又是审批人**（自批自核禁止）
   - 一个人可以同时是一级审批人和品控主管，但不能对自己上报的工单做任何审批操作
2. **品控主管权限边界**：
   - 品控主管可以对被暂扣的批次发起"快速放行"（误判复核）。
   - 品控主管不能审批物流类工单（物流类和品控类是两套审批人体系）。
   - 品控主管可以查看自己仓库下的所有工单（方便追责）。
3. **仓库级隔离**：如果涉及多仓库，用户只能看到自己仓库的运单/工单（通过 `warehouse_id` 字段过滤）。

## 六、V2 数据同步策略与一致性

### 假设设定

| 策略 | 说明 |
|------|------|
| 同步方式 | **混合模式**：上报异常时实时拉取 + 定时批量同步 |
| 实时拉取 | 每次上报异常/扫描操作时，调用 V2 接口获取最新运单信息 |
| 定时同步 | 每 30 分钟批量同步近期更新的运单到快照表 |
| 数据新鲜度标注 | 前端明确标注："实时获取自 V2" / "使用本地缓存，同步于 XX:XX" |

### 一致性边界处理

**场景：V2 运单信息在处理期间变更（如金额被更正）**

| 策略 | 说明 |
|------|------|
| 方案选择 | **关键节点实时校验**：审批操作每次打开工单详情时重新从 V2 拉取运单数据，发现金额/状态变化时标记差异 |
| 差异处理 | 若金额变更 → 在工单详情中标注"金额已变更（原 X 元 → 现 Y 元）"，不自动阻断审批；审批人自行判断 |
| 状态变更 | 若运单在 V2 端已标记为"已完成"/"已关闭" → 提示"该运单在 V2 端已结束处理，请确认是否仍需处理" |

### 选择依据

1. **实时性要求**：上报异常和审批是两个需要"最新数据"的关键环节，仅靠定时同步有滞后风险。
2. **性能平衡**：每次页面刷新都实时拉取会增加 V2 负担，混合模式在关键节点实时拉取、展示列表用缓存。
3. **30 分钟同步间隔**：能在 V2 恢复正常后半小时内完成增量同步，同时不造成过高负载。

## 七、品控暂扣超时时长

### 假设设定

| 参数 | 值 | 说明 |
|------|-----|------|
| 品控暂扣超时 | 4 小时 | 超时后强制升级到二级审批 |

### 设定依据

1. **独立于审批超时**：品控暂扣 ≠ 审批等待。品控暂扣意味着"货物被锁定在仓库，占着储位、无法出库、产生压仓成本"。审批超时可以给 48 小时，但货物不能被扣 48 小时不动。
2. **压仓成本**：
   - 假设仓库日租金 / 100 SKU ≈ 100 元，单 SKU 日均成本约 1 元。
   - 若扣留 4 小时成本 ≈ 0.17 元/SKU，仍可接受。
   - 若扣留 48 小时成本 ≈ 2 元/SKU × 100 SKU = 200 元，运营成本偏高。
3. **实际运营场景**：仓库通常几小时内就需要确定一批货是"放行"还是"退回"，4 小时足够品控人员核实一轮。如果 4 小时内无法决策，说明情况复杂，应直接升级处理。
4. **与审批超时的关系**：品控暂扣 4h << 审批 48h，确认为独立配置，不共用。

## 八、品控规则触发阈值

### 假设设定

| 规则编码 | 异常子类型 | 触发条件 | 严重度 | 默认审批层级 |
|----------|-----------|---------|--------|-------------|
| QC-QTY-01 | 数量不符 | 扫描数量与预期差异 ≥ 5% | minor（<20%差异）/ critical（≥20%差异） | 二级 |
| QC-DEF-01 | 外观破损 | 破损等级 ≥ 3 级（1-5 级） | major | 二级 |
| QC-DEF-02 | 外观破损 | 破损等级 = 1 或 2 级 | minor | 一级 |
| QC-SPEC-01 | 规格不符 | 规格标签与运单描述不匹配 | major | 二级 |
| QC-LBL-01 | 标签错误 | 条码不能识别 或 标签与货物不符 | critical | 二级 |
| QC-BAT-01 | 批次异常 | 批次号在系统内不存在 | critical | 二级 |
| QC-BAT-02 | 批次异常 | 批次对应商品与运单不一致 | critical | 二级 |

### 设定依据

1. **数量差异 5%**：参考零售/物流行业通行的"允许误差率"。5% 以内的差异可视为正常损耗或计数误差；超过则认定为系统性差异需处理。
2. **破损等级分界**：3 级破损通常意味着"货物功能受损但可修复"或"包装严重损坏但货物完好"，属于需要重点处理的程度。1-2 级（轻微划痕、包装褶皱）可放宽处理。
3. **规则可配置**：所有阈值均存储在 `qc_rules.condition_config` JSONB 字段中，后台可随时调整，严禁硬编码。

## 九、品控主管角色权限边界

### 假设设定

#### 品控主管的权限范围

| 能力 | 允许 | 说明 |
|------|------|------|
| 扫描操作 | ✓ | 日常也需要参与扫描 |
| 查看扫描记录 | ✓ | 仓库范围内全部 |
| 发起误判快速放行 | ✓ | 需填写复核原因，操作留痕 |
| 查看品控工单 | ✓ | 仓库范围内全部 |
| 审批物流工单 | ✗ | 品控主管只管理品控，不干预物流审批 |
| 审批品控工单 | ✗ | 品控主管可以"快速放行"(绕过审批)，但不能作为常规审批人 |
| 配置品控规则 | ✗ | 由管理员统一配置 |
| 配置审批分级 | ✗ | 由管理员统一配置 |

#### 与审批人权限的重叠处理

| 场景 | 处理方式 |
|------|---------|
| 某人同时是品控主管 + 一级审批人 | 对品控工单只能执行快速放行，不能执行常规审批 |
| 某人同时是品控主管 + 二级审批人 | 对品控工单只能执行快速放行，不能执行常规审批 |
| 品控主管上报了异常工单 | 不能对自己上报的工单做任何审批操作（包括快速放行） |
| 品控主管的快速放行权限 | 仅对"品控暂扣中"的批次有效，已进入审批流程的工单不能绕过审批 |

#### 快速放行的操作留痕

```typescript
interface FastReleaseRecord {
  ticketId: number;
  approverId: string;
  approverRole: 'qc_supervisor';
  reason: string;        // 复核原因（必填，不允许空字符串）
  evidence?: string;     // 复核证据（可选）
  releasedAt: Date;
}
```

### 设定依据

1. **权限隔离**：品控主管的"快速放行"是一个特殊的安全阀——允许在明确误判时快速修正，但必须与常规审批完全隔离，否则会破坏审批流程的权威性。
2. **不重叠审批**：品控主管不参与常规审批，避免"既是裁判又是运动员"的冲突。快速放行和常规审批是不同的操作路径，互不干扰。
3. **留痕要求**：每次快速放行都必须有文字说明+操作人记录，防止滥用。

## 十、向产品经理的问题清单（主动澄清意识）

> 以下问题在实际项目中"如果允许，我会向产品经理确认"，体现工程判断主动性：

1. **"品控暂扣超时 4 小时后，如果二级审批人也觉得不需要处理（属于误判），快速放行还能用吗？"** → 确认超时升级后是否还有"兜底的兜底"。
2. **"如果一笔运单同时有品控异常和物流异常（比如扫描时发现破损，出库后客户又报丢件），系统应该合并为一个工单还是分开？"** → 影响工单表设计（当前设计允许并存）。
3. **"V2 系统的用户权限体系是怎样的？V3 是否需要对接 V2 的登录/权限系统？还是 V3 独立管理用户？"** → 影响鉴权方案。
4. **"赔付金额的计算规则是什么？是按货值全额赔付还是按百分比？是否包含运费？"** → 影响赔付记录表的字段设计。
5. **"不同仓库的操作员是否能看到其他仓库的运单信息？"** → 影响多租户隔离的粒度。
6. **"品控主管可以发起快速放行，但放行后发现'其实是真异常'，这时怎么追责/回滚？"** → 影响操作留痕的详细程度。
7. **"工单超时自动驳回后，上报人还能重新提交完全相同的工单吗？还是需要走升级/申诉通道？"** → 影响拒绝→重提→超时路径的闭环。
8. **"V2 接口的 SLA 是怎样的？高并发时 V2 会不会限制 V3 的调用频率？"** → 影响重试策略和同步频率的设计。
9. **"如果有供应商直接联系品控主管要求放行被暂扣的批次，品控主管是否有权限越过系统流程直接操作？"** → 确认流程之外是否存在"线下通道"需要系统留痕。
10. **"品控类异常的'重新采购'是自动生成采购单还是仅作为建议推送给采购部门？"** → 影响执行联动的触发方式。
